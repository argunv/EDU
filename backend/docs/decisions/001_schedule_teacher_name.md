# ADR / Решение: teacher_name вместо teacher_id в расписании

## Контекст

В модели `ScheduleSlot` (расписание уроков) преподаватель задаётся полем `teacher_name` (строка), а не внешним ключом `teacher_id` на таблицу `users`.

## Решение

Хранить в слоте расписания строку с ФИО преподавателя (`teacher_name`). Связь «кто какой предмет ведёт» задаётся через `TeacherClass` и `TeacherSubject` (по `user_id`). В UI и при проверке конфликтов сопоставление «слот → учитель» идёт по совпадению `ScheduleSlot.teacher_name` с `User.name`.

## Причины

- Упрощение схемы и миграций: не требуется обязательная связь по FK для каждого слота.
- Удобное отображение в админке и отчётах без джойнов на users.
- Историчность: при удалении или смене учётки учителя старые слоты остаются читаемыми по имени.

## Последствия

- При смене ФИО учителя в профиле существующие слоты расписания не обновляются автоматически. Требуется либо ручное переназначение слотов, либо синхронизация имени при сохранении пользователя (если такой сценарий будет реализован).
- Проверка конфликтов «один учитель — два класса в одно время» опирается на совпадение имени; дубликаты имён в системе могут дать ложные конфликты.

## Альтернативы

- **teacher_id (FK на users):** автоматическое обновление при смене имени и однозначная привязка; усложнение миграций и необходимость каскадов/значений по умолчанию при удалении учителя.
- Оставлен текущий вариант как компромисс для текущего масштаба продукта.
